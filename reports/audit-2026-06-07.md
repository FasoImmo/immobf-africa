## Note méthodologique — headers de sécurité

L'outil de récupération web disponible cette semaine (`web_fetch`) ne restitue que le contenu rendu et les balises `<meta>`, pas les en-têtes de réponse HTTP bruts (`Content-Security-Policy`, `Strict-Transport-Security`, `X-Frame-Options`, etc.). L'accès direct au domaine via le shell sandboxé est bloqué par la liste blanche réseau (`blocked-by-allowlist`), et l'extension Chrome n'était pas connectée pour inspecter le trafic réseau. La vérification des en-têtes de sécurité s'appuie donc cette semaine **uniquement sur l'inspection du code source** (`frontend-web/next.config.js`), qui montre la configuration suivante dans `headers()` appliquée à `/:path*` :

- `Content-Security-Policy` → ✅ présent (politique complète : `default-src 'self'`, `frame-ancestors 'none'`, `upgrade-insecure-requests`, etc.)
- `Strict-Transport-Security` → ✅ présent (`max-age=63072000; includeSubDomains; preload`)
- `X-Frame-Options` → ✅ présent (`DENY`)
- `X-Content-Type-Options` → ✅ présent (`nosniff`)
- `Referrer-Policy` → ✅ présent (`strict-origin-when-cross-origin`)
- `Permissions-Policy` → ✅ présent (caméra, micro, géoloc, paiement, USB, FLoC tous restreints)

Sous réserve que le déploiement Vercel reflète bien ce fichier (ce qui est l'hypothèse normale pour une plateforme Next.js déployée depuis ce dépôt), les 6 en-têtes sont correctement configurés. **Recommandation** : confirmer cela en direct la semaine prochaine via un outil exposant les en-têtes bruts (Chrome DevTools / extension connectée, ou service de scan d'en-têtes).
