## Note méthodologique — headers de sécurité

L'outil de récupération web disponible (`web_fetch`) ne restitue que le contenu rendu et les balises `<meta>`, pas les en-têtes de réponse HTTP bruts (`Content-Security-Policy`, `Strict-Transport-Security`, `X-Frame-Options`, etc.). La vérification s'appuie donc uniquement sur l'inspection du code source (`frontend-web/next.config.js`), qui confirme la présence de :

- `Content-Security-Policy` → ✅ politique complète (`default-src 'self'`, `frame-ancestors 'none'`, `upgrade-insecure-requests`, etc.)
- `Strict-Transport-Security` → ✅ `max-age=63072000; includeSubDomains; preload`
- `X-Frame-Options` → ✅ `DENY`
- `X-Content-Type-Options` → ✅ `nosniff`
- `Referrer-Policy` → ✅ `strict-origin-when-cross-origin`
- `Permissions-Policy` → ✅ caméra, micro, géoloc, paiement, USB et FLoC restreints

Sous réserve que le déploiement Vercel reflète bien ce fichier, les 6 en-têtes sont correctement configurés. Cette limitation est inchangée depuis le rapport du 2026-06-07.

## Détail SEO

- `robots.txt` : la requête retourne un corps vide (aucun contenu), sans titre ni texte exploitable — le fichier semble absent ou mal configuré.
- `sitemap.xml` : même constat, corps vide.

Ces deux éléments n'ont pas pu être confirmés comme erreurs HTTP explicites (l'outil de fetch ne renvoie pas le code de statut brut), mais l'absence totale de contenu sur des URLs qui devraient renvoyer du texte/XML est un signal fort d'absence ou de mauvaise configuration. Impact : référencement organique probablement pénalisé (les moteurs de recherche n'ont pas d'instructions de crawl ni de plan du site).

## Détail bannière cookies

La page `/legal/cookies` décrit un mécanisme de consentement (boutons « Accepter les cookies nécessaires » / « Refuser les cookies non essentiels », cookie `immobf_cookie_consent`). Cependant, le rendu HTML récupéré de la page d'accueil ne contient aucun élément de bannière de consentement visible. Deux explications possibles : la bannière est injectée dynamiquement en JavaScript après le chargement (non visible avec l'outil de fetch actuel, qui ne rend pas le JS), ou elle est absente. Recommandation : vérification visuelle directe (navigateur) pour confirmer son affichage réel aux nouveaux visiteurs.

## Détail dépendances (npm audit --omit=dev)

**Backend** (631 dépendances analysées) — 7 vulnérabilités : 3 élevées, 4 modérées.
- Élevée : `@mapbox/node-pre-gyp` (≤1.0.11) dépend d'une version vulnérable de `tar` → impacte `argon2` (0.27.2–0.31.2), utilisé pour le hachage des mots de passe. Risque lié à la chaîne de build native, pas à l'algorithme Argon2 lui-même.
- Modérée : `express` (4.21.0–4.22.1) dépend d'une version vulnérable de `qs`.
- Modérée : `uuid` — dépassement de tampon possible dans `v3/v5/v6` si un buffer est fourni en argument (correctif disponible en v14, changement majeur).
- Correctifs disponibles via `npm audit fix` / `npm audit fix --force` (ce dernier impliquant des montées de version majeures).

**Frontend** (748 dépendances analysées) — 3 vulnérabilités : 2 élevées, 1 modérée.
- Modérée : `postcss` (<8.5.10) — XSS possible via balise `</style>` non échappée dans la sortie CSS stringifiée, affecte `next`.
- 2 élevées (détail non affiché par `npm audit` en sortie résumée — nécessite `npm audit` complet pour le détail des CVE).
- Correctif disponible via mise à niveau majeure de `next` (16.2.9).
