## Note méthodologique — headers de sécurité et SEO

L'outil `web_fetch` disponible ne restitue que le contenu rendu et les balises `<meta>`, pas les en-têtes de réponse HTTP bruts. Il ne renvoie également aucun contenu exploitable pour les ressources non-HTML (`robots.txt`, `sitemap.xml` reviennent vides, sans indication de code de statut), que le fichier soit présent ou non — cette limitation est confirmée par contraste avec `/api/v1/health` (JSON, contenu bien restitué) et les pages HTML (contenu bien restitué). Il s'agit donc d'une limitation de l'outil sur les types de contenu texte brut/XML, pas d'un signal d'absence.

Pour cette raison, les vérifications « Headers sécurité » et « SEO » s'appuient sur l'inspection directe du code source :

- `frontend-web/next.config.js` → `headers()` confirme la présence de `Content-Security-Policy`, `Strict-Transport-Security`, `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`.
- `frontend-web/public/robots.txt` existe avec un contenu valide (`User-agent: *`, `Allow: /`, exclusions `/admin`, `/account`, `/payment/callback`, `/api/`, référence au sitemap).
- `frontend-web/pages/sitemap.xml.js` existe : route dynamique (pages router) qui génère un XML valide avec pages statiques + annonces publiées, avec dégradation gracieuse si l'API est indisponible.

**Correction par rapport au rapport du 2026-06-17** : ce dernier concluait à une « absence probable » de `robots.txt`/`sitemap.xml` sur la seule base du fetch vide. L'inspection du code cette semaine montre que les deux fichiers/routes existent et semblent correctement implémentés. Sous réserve que le déploiement Vercel reflète bien ce code, le score SEO est revu à la hausse. Une vérification en direct (navigateur ou outil exposant le code de statut HTTP brut) reste nécessaire pour lever le doute définitivement.

## Détail bannière cookies

Le composant `components/CookieBanner.js` est importé et monté dans `pages/_app.js`, donc rendu sur l'ensemble des pages de l'application côté client. Son absence du HTML statique récupéré par `web_fetch` (qui n'exécute pas de JavaScript) est attendue et ne constitue pas une anomalie. Contrairement au rapport du 2026-06-17 qui notait cela en avertissement ⚠️, l'inspection du code permet cette semaine de lever ce doute : le mécanisme est bien implémenté.

## Détail i18n / locale anglaise

La page `/en` répond `200` et redirige les liens internes vers les chemins préfixés `/en/...` (ex. `/en/login`, `/en/legal`). Toutefois, le contenu textuel rendu (titre « ImmoBF Africa — accueil », « Nous optimisons votre choix », « S'inscrire ») reste en français malgré le préfixe de locale anglaise. À vérifier : soit le contenu n'est pas encore traduit pour le segment testé, soit la traduction ne s'applique pas correctement à ce rendu côté serveur.

## Détail dépendances (npm audit --omit=dev)

**Backend** — 21 vulnérabilités, toutes de sévérité modérée (0 élevée, 0 critique) :
- Chaîne `@opentelemetry/core` / `@opentelemetry/resources` / `@opentelemetry/sdk-trace-base` vulnérables, propagées via `@prisma/instrumentation`, `@opentelemetry/instrumentation-mysql2` et `@sentry/opentelemetry`.
- `uuid` (<11.1.1) : vérification de limites de buffer manquante dans v3/v5/v6 lorsqu'un buffer est fourni — propagé via `node-cron`. Correctif disponible (`uuid@14.0.1`, changement majeur).
- Correctifs partiels via `npm audit fix` ; correction complète nécessite `npm audit fix --force` (changements majeurs).

**Frontend** — 3 vulnérabilités (2 élevées, 1 modérée) :
- Élevée : `form-data` — injection CRLF via noms de champs/fichiers multipart non échappés. Correctif disponible sans changement majeur (`npm audit fix`).
- Élevée : `next` (9.3.4-canary.0 – 16.3.0-canary.5) — multiples CVE incluant déni de service, smuggling de requêtes HTTP dans les rewrites, contournement de Middleware/Proxy en Pages Router avec i18n (pertinent ici, l'app utilise `i18n` dans `next.config.js`), et empoisonnement de cache. Correctif nécessite mise à niveau majeure vers `next@16.2.9`.
- Modérée : `postcss` (<8.5.10) — XSS via `</style>` non échappé, dépendance de `next`.
